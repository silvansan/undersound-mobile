# UnderSound Mobile App Plan

This document is written for Codex / AI coding agents so they can understand the intended mobile app architecture before generating code.

## 1. Project Summary

**App name:** UnderSound  
**Main goal:** A mobile listener app for the UnderSound live translation / live audio system.

The app should let people scan a QR code at an event, open the correct UnderSound server/event, choose an audio channel/language, and listen with the phone screen off or locked.

The key reason for the mobile app is that normal browser/WebRTC listening can stop or become unreliable when Android/iOS users lock the screen or put the phone in their pocket. The mobile app must behave more like a real audio/radio app.

## 2. Current Server Context

UnderSound currently uses:

- Docker / Portainer deployment
- Node.js frontend/backend web app
- LiveKit for WebRTC audio rooms
- Redis for LiveKit
- LiveKit Egress for HLS output
- HLS output mounted into the web app under `/public/hls`
- Public web access through Nginx Proxy Manager
- Existing public links should remain stable

Current deployment idea:

- Web app public URL, for example: `https://voice.example.com`
- LiveKit WebSocket URL, for example: `wss://voice.example.com`
- Listener pages can be opened by QR code
- HLS stream URLs should be exposed per event/channel for mobile playback

## 3. Mobile App Technology Choice

Use **Flutter** for the mobile app.

Target platforms:

- Android first
- iOS second
- Later: optional web/desktop if useful

Reason:

- One codebase for Android and iOS
- Good UI speed
- Good packages for background audio
- Easier than writing separate native Android + iOS apps

Recommended Flutter packages:

- `just_audio` for audio playback
- `audio_service` for background audio service and lock-screen/media notification integration
- `audio_session` for audio focus/session configuration
- `mobile_scanner` or similar for QR code scanning
- `http` or `dio` for API calls
- `shared_preferences` or `flutter_secure_storage` for saving last server/event

## 4. Core Requirement: Background Audio

The app must keep playing audio when:

- The screen is turned off
- The phone is locked
- The user switches to another app
- The phone is in the pocket
- Bluetooth headphones are connected
- The user uses lock-screen controls

This means the app must not be built like a simple foreground-only player.

Implementation expectation:

- Use `audio_service` as the background audio layer.
- Use `just_audio` as the actual stream player.
- Use `audio_session` to configure audio focus properly.
- Android must use foreground service/media notification behavior.
- iOS must enable background audio mode.

Do not build the first mobile app around WebRTC playback if the main listener use case is locked-screen listening. Prefer HLS/audio stream playback for listeners.

## 5. Audio Transport Strategy

The mobile app should primarily consume **HLS audio streams** generated by UnderSound.

Why:

- HLS is easier to play as a normal background media stream.
- HLS works well with audio player libraries.
- Background playback is more predictable than browser WebRTC.
- It is suitable for listener-only users.

Expected tradeoff:

- HLS has more latency than WebRTC.
- This is acceptable for “phone in pocket” listeners.
- Speakers/interpreters/admins may still use WebRTC in the browser or a future dedicated app mode.

Possible future modes:

1. **Listener Mode**
   - HLS playback
   - Background audio
   - Lock screen controls
   - Main mobile app priority

2. **Interpreter/Speaker Mode**
   - WebRTC publishing/listening
   - Lower latency
   - Requires foreground app
   - Later phase

## 6. QR Code Flow

The user journey should be:

1. User opens the UnderSound mobile app.
2. User taps “Scan QR”.
3. QR code contains a deep link or normal URL, for example:

   ```text
   https://voice.example.com/e/event-id
   ```

   or:

   ```text
   undersound://join?server=https://voice.example.com&event=event-id
   ```

4. App parses:
   - server URL
   - event ID
   - optional channel ID
   - optional access token/password
5. App calls the UnderSound API to load event data.
6. App shows available channels/languages.
7. User taps a channel.
8. App starts HLS playback.
9. User can lock phone and keep listening.

## 7. API Expectations

The mobile app should not hard-code channels. It should discover them from the UnderSound server.

Suggested API endpoints for the server:

### Get event info

```http
GET /api/mobile/events/:eventId
```

Example response:

```json
{
  "id": "event-id",
  "name": "Sunday Service",
  "status": "live",
  "serverTime": "2026-05-11T14:00:00Z",
  "channels": [
    {
      "id": "fr",
      "name": "French",
      "languageCode": "fr",
      "isLive": true,
      "hlsUrl": "https://voice.example.com/hls/event-id/fr/index.m3u8"
    },
    {
      "id": "en",
      "name": "English",
      "languageCode": "en",
      "isLive": true,
      "hlsUrl": "https://voice.example.com/hls/event-id/en/index.m3u8"
    }
  ]
}
```

### Optional: validate access

```http
POST /api/mobile/events/:eventId/access
```

Request:

```json
{
  "password": "optional-event-password"
}
```

Response:

```json
{
  "ok": true,
  "token": "short-lived-listener-token"
}
```

### Optional: channel status polling

```http
GET /api/mobile/events/:eventId/status
```

Response:

```json
{
  "eventId": "event-id",
  "status": "live",
  "channels": [
    {
      "id": "fr",
      "isLive": true
    }
  ]
}
```

## 8. App Screens

Keep the first version simple.

### Screen 1: Welcome / Connect

Purpose:

- Scan QR
- Enter server URL manually
- Reopen last event

Elements:

- UnderSound logo/name
- “Scan QR Code” button
- “Enter link manually” button
- “Recent event” if available

### Screen 2: Event Overview

Purpose:

- Show event name
- Show available channels
- Show live/offline state

Elements:

- Event title
- Server URL
- Channel cards:
  - language name
  - live/offline badge
  - tap to listen

### Screen 3: Player

Purpose:

- Play selected channel

Elements:

- Event name
- Selected channel/language
- Play/pause button
- Reconnect button
- Change channel button
- Connection state:
  - Connecting
  - Playing
  - Buffering
  - Offline
  - Error
- Optional latency note: “Low / Normal / Delayed”

### Screen 4: Settings

Purpose:

- Remember last server
- Clear saved data
- Diagnostics

Elements:

- Server URL
- App version
- Clear cache
- Show stream URL for debugging
- Optional log export

## 9. State Management

- `Riverpod` or `Provider`

## 10. Deep Link Support

Support both:

1. Normal HTTPS links:

```text
https://voice.example.com/e/event-id
```

2. Custom scheme:

```text
undersound://join?server=https://voice.example.com&event=event-id
```

First version can start with QR scanning only. Deep linking can be added after the app works.

## 11. Error Handling

The app must be friendly for non-technical users.

Examples:

- If server cannot be reached:
  - “Could not reach the UnderSound server.”
  - Show retry button.
- If event does not exist:
  - “This event link is not valid.”
- If channel is offline:
  - “This channel is not live yet.”
- If audio fails:
  - “Audio stream not available. Try another channel or reconnect.”
- If password is required:
  - Ask for password before showing channels.

Do not show raw stack traces to users.

## 15. Important Do / Don’t Rules for Codex

### Do

- Build a clean Flutter app.
- Keep screens simple and readable.
- Use HLS for listener playback.
- Use background audio architecture from the beginning.
- Keep all server URLs configurable.
- Store recent server/event locally.
- Write code in small, understandable files.
- Add comments where mobile background audio setup is tricky.
- Keep the app self-hosted friendly.
- Assume users may deploy UnderSound on their own domain.

### Don’t

- Do not hard-code `voice.example.com`.
- Do not hard-code one channel/language.
- Do not rely on WebRTC for background listener playback.
- Do not require Firebase.
- Do not require a central cloud server.
- Do not break self-hosting.
- Do not build complex login first.
- Do not make the app dependent on Google services unless absolutely necessary.
- Do not assume the public app store is required for the first test version.
- Do not rename the existing server/API project automatically.

## 18. Open Questions

These are decisions the human maintainer should confirm:

1. Should the mobile app live in the same GitHub repo as UnderSound or in a separate repo?
2. Should event links use `/e/:eventId`, `/join/:eventId`, or another route?
3. Should listener access be public, password-protected, or token-based?
4. Should the app show all events or only events opened by QR?
5. Should the app eventually include speaker/interpreter mode?

## 19. Recommended Repo Strategy

Prefer a separate repo:

```text
UnderSound-Mobile
```

Reason:

- Flutter has a very different project structure from the Docker/web/server app.
- Mobile CI/CD is separate.
- Releases are separate.
- Easier for Codex to understand the mobile scope.

But the server repo should contain a short link to the mobile repo once it exists.

## 20. Final Target Experience

A church/event visitor should be able to:

1. Install/open UnderSound Mobile.
2. Scan a QR code.
3. Tap “English”, “French”, “German”, etc.
4. Put the phone in their pocket.
5. Keep listening reliably.
